# AutoWrite 使用说明

## 1. 这个工具是干什么的

AutoWrite 是一个本地小说自动写作试验工具。它会把一个小说点子拆成大纲，再按章节生成正文，并在生成过程中做逻辑审稿、文风检查、AI 痕迹检查和连续性记录。

它适合用来试验：

- 一万五千字左右短篇。
- 5-7 章完整短篇，也可用 3 章小样试跑。
- 网文套路样稿。
- 点子批量试跑。
- 大纲到正文的自动流水线效果。

它暂时不接入你的现有生产线：

- 不接入短篇生产线。
- 不接入游戏制作流程。
- 不接入公文/PPT/个人知识库。
- 不自动改任何全局 Codex 规则。

## 2. 安装位置

工具目录：

```text
D:\文档\工具\AutoWrite
```

常用文件：

- `启动AutoWrite-本地安全.bat`：推荐启动入口。
- `.env.example`：配置模板。
- `.env`：你自己创建的本地密钥配置文件。
- `Novel\`：生成的小说正文输出目录。
- `Outline\`：生成的大纲输出目录。
- `Role\`：写作、审稿、责编等提示词。
- `material_library.json`：素材库。
- `pattern_library.json`：套路库。

## 3. 第一次使用前配置

先在 `D:\文档\工具\AutoWrite` 里复制一份：

```text
.env.example -> .env
```

然后打开 `.env`，填写这几个字段：

```env
OPENAI_API_KEY="你的 API Key"
OPENAI_BASE_URL="你的 OpenAI 兼容接口地址"
OPENAI_MODEL="你的模型名"

DEFAULT_CHAPTERS=6
DEFAULT_WORDS_PER_CHAPTER=2500
WEB_PORT=18080
MODEL_TIMEOUT_SECONDS=180
MODEL_MAX_RETRIES=5
APP_INVOKE_ATTEMPTS=3
```

注意：

- `OPENAI_MODEL` 必须写服务商真实支持的模型名。
- 如果不知道模型名，不要乱填，先去你使用的 API 平台复制。
- `WEB_PORT=18080` 是为了避开常见的 8080 端口冲突。
- `.env` 只保存在本机，不要发给别人。

## 4. 启动方式

双击：

```text
启动AutoWrite-本地安全.bat
```

启动后浏览器会打开：

```text
http://127.0.0.1:18080
```

如果浏览器没自动打开，就手动复制上面的地址到浏览器。

关闭方式：

- 关闭启动脚本弹出的命令行窗口。
- 或在命令行窗口里按 `Ctrl+C`。

## 5. 推荐第一次试跑

第一次不要直接跑一万字，先跑一个 3 章小样，确认模型、接口、流程都通。

建议配置：

- 章节数：`3`
- 每章字数：`1200` 或 `1500`
- 写手风格：`默认自然`
- 创作套路：先选普通套路或 `无固定主套路`
- 素材：抽取 4 项即可

确认能稳定跑完后，再试：

- 6 章 × 2500 字，约 15000 字，推荐作为质量短篇默认配置。
- 5 章 × 3000 字，约 15000 字，适合更厚的场景化短篇。
- 3 章 × 1500 字，约 4500 字，适合卡点小样。

## 6. 新建小说的基本流程

### 6.1 输入点子

在页面左侧输入你的小说灵感。

写法可以很粗：

```text
女主重生回到被退婚当天，这次她不解释，直接借退婚宴让男主家族露出财务黑洞。
```

也可以写得更明确：

```text
现代豪门短篇，女主前世替男主背锅坐牢，重生回订婚宴。
目标是 6 章，每章 2500 字，第 2 章出现第一次大反转，第 4 章彻底切断旧感情，第 6 章收束。
风格冷静克制，不要霸总油腻感。
```

### 6.2 灵感精炼

如果点子很粗，可以点“灵感精炼”。

它会问 1-2 个问题，帮你把主角身份、核心冲突、爽点、世界观基调补清楚。

如果你已经想清楚了，可以跳过。

### 6.3 设置章节和字数

常用组合：

- 快速试跑：`3 章 × 1200-1500 字`
- 质量短篇：`6 章 × 2500 字`
- 更厚场景：`5 章 × 3000 字`
- 卡点测试：`3 章 × 1500 字`

字数是软目标，模型可能多写或少写。不要把它当排版软件。

### 6.4 选择写手风格

当前内置风格包括：

- 默认自然
- 热血爽文
- 文艺细腻
- 冷峻纪实
- 轻松幽默
- 成人情感
- 悬疑压迫
- 情感拉扯
- 甜宠轻喜
- 古言雅致
- 现实群像
- 商战职场

建议：

- 女频情感短篇：优先 `情感拉扯`、`冷峻纪实`、`默认自然`。
- 爽文短篇：优先 `热血爽文`、`默认自然`。
- 悬疑/规则怪谈：优先 `悬疑压迫`、`冷峻纪实`。
- 职场商战：优先 `商战职场`、`冷峻纪实`。

### 6.5 选择创作套路

套路分两类：

- 普通套路：更自由，适合第一次试。
- 强套路：会生成“套路契约”，对节奏、反转、结局要求更硬。

强套路更适合商业短篇，但也更容易因为风格不兼容、大纲不满足契约而报错。

第一次建议先不用强套路。等流程跑通后，再试：

- 女频虐恋觉醒
- 真假千金身份逆袭
- 古言宅斗复仇
- 规则怪谈
- 男频升级打脸

如果页面提示“当前写手风格与所选套路不兼容”，就按提示换风格。

### 6.6 抽取素材

素材库会从这些大类里抽东西：

- 世界舞台
- 主角人设
- 配角关系
- 金手指与能力
- 剧情事件
- 核心冲突
- 职业与资源
- 场景与氛围

你可以先随机抽取。看到不合适的素材，再重新抽。

素材只是辅助，不要让随机素材压过你的核心点子。

### 6.7 启动流水线

点击启动后，它会先生成大纲。

这一步是“架构师”工作：

- 书名
- 世界观
- 分章细纲
- 每章冲突和钩子

大纲生成后会停下来等你审批。

### 6.8 审批大纲

你要认真看大纲，尤其看：

- 前三章有没有进入冲突。
- 中段有没有反转或升级。
- 结尾有没有收束。
- 人物动机是否成立。
- 有没有跑偏到你不想写的题材。

如果大纲明显不对，不要批准，重新改点子再跑。

如果大纲基本可用，再批准继续。

### 6.9 自动写正文

批准后，工具会逐章生成正文。

每章大致经过：

```text
写手生成 -> 审计检查 -> 责编检查 -> 连续性检查 -> 不合格退回重写 -> 合格后入库
```

页面右侧会显示日志。

如果它反复重写，通常说明：

- 大纲要求太多。
- 字数太少。
- 套路契约太硬。
- 模型能力不够稳定。
- API 超时或返回格式不稳定。

## 7. 输出在哪里

生成结果主要看两个目录：

```text
D:\文档\工具\AutoWrite\Outline
D:\文档\工具\AutoWrite\Novel
```

`Outline` 里是大纲 JSON。

`Novel` 里是小说正文 TXT。

如果你后面想让我帮你评估某篇输出，把对应 TXT 或 JSON 路径发给我即可。

### 7.1 小说 Tag 与词库维护

新生成的大纲会在 JSON 顶层增加 `novel_tags`：

```json
{
  "novel_tags": {
    "core": "规则怪谈",
    "情节": ["女性成长", "无限流"],
    "角色": ["大女主"],
    "情绪": ["惊悚"],
    "背景": ["副本世界"]
  }
}
```

规则是：

- `core` 必须且只能有 1 个核心 Tag。
- 情节、角色、情绪、背景四类辅助 Tag 合计必须为 5-7 个，各分类可以为空。
- 所有词必须逐字来自项目根目录的 `novel_tag_library.json`，模型不能自由创造新词。
- Tag 只用于投稿或上传网站时参考，不会传给写手改变正文。
- 没有 `novel_tags` 的旧 schema v2 大纲仍可正常加载，页面会显示“未打标”。

维护词库时，直接编辑 `novel_tag_library.json` 中的字符串数组。保存后无需重启，下一篇大纲会读取最新内容。四个辅助分类名不能改动；所有 Tag 必须是非空字符串，整个词库内不能重名，辅助词总数不能少于 5 个。

如果词库 JSON 损坏、分类缺失或出现重复词，系统会在调用模型前报告“小说Tag库校验失败”。如果模型选了库外词、放错分类、选少于 5 个或多于 7 个，架构师会按现有重试次数重新生成；连续失败后会终止并显示具体 Tag 问题。

## 8. 洗文/二次改写功能怎么理解

工具里有基于已有大纲继续改写的功能。

建议只用于：

- 你自己生成的大纲。
- 你自己的旧稿。
- 已授权或可合法使用的材料。
- 只提炼结构、不照搬表达的练习材料。

不要直接复制别人的完整小说、受版权保护文本或平台付费内容进去让它改写。

更稳妥的做法是：

1. 先让 Codex 或你自己把参考文拆成“结构模板”。
2. 只保留节奏、情绪曲线、卡点位置、反转类型。
3. 再用 AutoWrite 跑一个全新人物、全新事件、全新表达的故事。

## 9. 批量点子功能

`BatchIdeaLauncher` 是高级功能，适合一次跑多个点子。

第一次不要用它。等你熟悉 Web 版后再试。

它需要单独配置：

```text
D:\文档\工具\AutoWrite\BatchIdeaLauncher\.env
```

主要字段：

```env
LAUNCHER_API_KEY="你的选择器模型 API Key"
LAUNCHER_BASE_URL="你的选择器模型接口"
LAUNCHER_MODEL="你的选择器模型名"
```

批量功能适合：

- 一次测试 10 个短篇点子。
- 自动给每个点子选套路和素材。
- 失败后单独重试。

不适合：

- 第一次配置。
- API 不稳定时。
- 还没明确平台方向时。

## 10. 常见问题

### 10.1 双击启动后闪退

常见原因：

- 没有 `.env`。
- `.env` 里没填 `OPENAI_API_KEY`。
- 虚拟环境 `.venv` 不存在。

先看命令行窗口里的提示。

### 10.2 浏览器打不开

手动访问：

```text
http://127.0.0.1:18080
```

如果打不开，检查启动窗口有没有报错。

### 10.3 端口被占用

改 `.env`：

```env
WEB_PORT=18081
```

然后重新双击启动脚本。

### 10.4 提示模型不存在

说明 `OPENAI_MODEL` 写错了，或者你的 API 平台不支持这个模型。

去 API 平台复制正确模型名。

### 10.5 一直超时

可以把 `.env` 里的超时时间调大：

```env
MODEL_TIMEOUT_SECONDS=300
MODEL_MAX_RETRIES=5
APP_INVOKE_ATTEMPTS=3
```

如果仍然超时，先降低章节数或每章字数。

### 10.6 输出太短

处理方式：

- 每章目标字数调高。
- 大纲里明确每章必须发生的事件。
- 不要让单章同时承载太多情节。
- 尽量先跑 3 章测试模型稳定性。

### 10.7 输出太 AI

先尝试：

- 换 `冷峻纪实` 或 `默认自然`。
- 不要选择过于夸张的套路。
- 在点子里明确“不要空泛抒情、不要总结大道理、不要库存表达”。

但注意：工具里的 AI 痕迹检查只是编辑风险提示，不保证通过平台 AI 检测。

### 10.8 卡在大纲审批

这是正常流程。它生成大纲后会等待你批准。

你要在页面上点批准，才会继续写正文。

## 11. 推荐试用路线

第一轮：验证能不能跑通。

```text
3 章 × 1200 字
默认自然
无固定主套路
随机素材 4 项
```

第二轮：验证商业套路。

```text
3 章 × 1500 字
情感拉扯
女频虐恋觉醒 或 真假千金身份逆袭
确认强套路契约
```

第三轮：跑质量短篇。

```text
6 章 × 2500 字
明确第 2 章卡点、第 4 章反转、第 6 章收束
```

## 12. 重要边界

- 不要把它生成的内容直接当终稿。
- 不要相信它能自动通过编辑审核。
- 不要相信它能自动通过 AI 检测。
- 不要用它处理涉密、公文、真实个人隐私或敏感材料。
- 不要直接改写未授权版权文本。
- 它适合做“第一版草稿”和“结构试验”，真正投稿前仍需要人工重写、删套话、补细节、查逻辑。
